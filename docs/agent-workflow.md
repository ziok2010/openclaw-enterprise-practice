
# 智能体协作流程设计

## 一、设计思路演进

### 阶段 1：一个"超级 AI"什么都干

**最初的尝试**：
- 一个 AI 智能体负责所有工作：需求、设计、开发、测试、部署
- 配置所有技能和工具

**问题**：
- 什么都干不好，生成质量不稳定
- 上下文混乱，经常忘记前面的内容
- 出错后难以定位问题

---

### 阶段 2：9 个专业智能体各司其职

**改进方案**：
- 按照真实团队的角色分工设计智能体
- 每个智能体专注一个领域
- 通过消息传递和记忆同步实现协作

**效果**：
- 生成质量显著提升
- 问题定位更容易
- 人机协作边界更清晰

---

## 三、9 个专业智能体职责设计

### 1. PM（产品经理智能体）

**核心职责**：
- 需求分析与 PRD 文档编写
- 字段定义、验收标准、业务流程
- 需求变更管理

**输入**：
- 业务需求（来自业务方或 Matrix 房间讨论）

**输出**：
- PRD 文档（Markdown 格式）
- 字段定义表
- 业务流程图（Mermaid）
- 验收标准清单

**交接对象**：PMO

**记忆配置**：
- 需求文档模板
- 字段定义规范
- 业务流程规范

**人工审核点**：
- PRD 输出后必须产品经理（人）审核

---

### 2. PMO（项目管理智能体）

**核心职责**：
- 任务拆解（Epic → Story → Task）
- 任务分配（给 Frontend、Backend、UI）
- 进度跟踪与预警

**输入**：
- PRD 文档

**输出**：
- 任务列表（Jira/Matrix）
- 时间表（Gantt 图）
- 风险预警

**交接对象**：Architect

**记忆配置**：
- 任务拆解规则
- 任务优先级标准
- 常见风险库

**人工审核点**：
- 无需人工审核，自动流转到 Architect

---

### 3. Architect（架构师智能体）

**核心职责**：
- 系统架构设计（模块划分、部署拓扑）
- 接口契约定义（API 形态、数据流）
- 技术选型建议

**输入**：
- 任务列表

**输出**：
- 架构设计文档
- 接口文档（OpenAPI 3.0）
- 技术选型说明

**交接对象**：UI、Frontend、Backend

**记忆配置**：
- 企业技术栈规范
- 架构设计模板
- 接口设计规范

**人工审核点**：
- 架构设计必须技术负责人（人）审核

---

### 4. UI（UI 设计智能体）

**核心职责**：
- 视觉设计（配色、排版、图标）
- 交互原型（Figma 集成）
- 设计规范输出

**输入**：
- 架构设计文档

**输出**：
- 设计规范文档
- 设计稿链接（Figma）
- 交互说明

**交接对象**：Frontend

**记忆配置**：
- 企业设计规范
- 配色方案库
- 组件库

**人工审核点**：
- 设计稿必须 UI 设计师（人）审核

---

### 5. Frontend（前端开发智能体）

**核心职责**：
- 前端代码生成（Vue3、React）
- 调用后端接口
- 前端性能优化

**输入**：
- 设计稿
- 接口文档

**输出**：
- 前端代码（GitLab）
- 组件文档
- 性能优化报告

**交接对象**：Backend（联调）

**记忆配置**：
- 前端编码规范
- 组件库使用规范
- 性能优化最佳实践

**人工审核点**：
- 代码提交后自动触发代码审查，人工无需主动审核

---

### 6. Backend（后端开发智能体）

**核心职责**：
- 后端代码生成（Java、Python、Go）
- 数据库设计
- 接口实现

**输入**：
- 接口文档

**输出**：
- 后端代码（GitLab）
- 数据库设计文档
- 接口实现说明

**交接对象**：Frontend（联调）

**记忆配置**：
- 后端编码规范
- 数据库设计规范
- 事务处理规范

**人工审核点**：
- 代码提交后自动触发代码审查，人工无需主动审核

---

### 7. Security（安全工程师智能体）

**核心职责**：
- 代码安全审计（SQL 注入、XSS、CSRF、权限漏洞）
- 敏感信息检测（密码、密钥泄露）
- 安全修复建议

**输入**：
- 前后端代码

**输出**：
- 安全审计报告
- 漏洞清单（按严重等级分类）
- 修复建议

**交接对象**：QA

**记忆配置**：
- OWASP Top 10 常见漏洞
- 企业安全规范
- 安全编码最佳实践

**人工审核点**：
- 高危漏洞必须修复后才能继续
- 低危漏洞可以标记为"已知风险"继续

---

### 8. QA（测试工程师智能体）

**核心职责**：
- 测试用例自动设计（功能、接口、性能）
- 自动化测试执行（Jenkins、Postman 集成）
- 缺陷分析和修复建议
- 测试报告生成

**输入**：
- 安全通过的代码

**输出**：
- 测试用例库
- 测试报告
- 缺陷清单

**交接对象**：Ops

**记忆配置**：
- 测试用例模板
- 测试规范
- 常见缺陷库

**人工审核点**：
- 测试失败必须返工
- 测试通过后自动流转到 Ops

---

### 9. Ops（运维工程师智能体）

**核心职责**：
- 服务器资源监控（Prometheus 集成）
- Docker/K8s 容器管理
- 自动部署
- 故障自动诊断和修复

**输入**：
- 测试通过的代码

**输出**：
- 部署结果
- 监控面板
- 部署文档

**交接对象**：Delivery

**记忆配置**：
- 部署流程
- 故障解决方案库
- 监控告警规则

**人工审核点**：
- 生产环境部署必须运维负责人（人）审批
- 测试环境部署可以自动进行

---

### 10. Delivery（交付经理智能体）

**核心职责**：
- 交付文档自动生成（部署文档、用户手册、API 文档）
- 版本说明编写
- 变更记录整理

**输入**：
- 部署结果

**输出**：
- 交付文档包
- 版本说明
- 变更记录

**交接对象**：客户

**记忆配置**：
- 交付文档模板
- 版本命名规范
- 变更记录规范

**人工审核点**：
- 版本发布必须项目经理（人）确认
- 文档生成后可自动同步到知识库

## 四、智能体协作关键设计

### 1. 智能体路由策略

**基于项目 ID 和阶段路由**：

```yaml
# 路由规则示例
routing:
  strategy: project-based
  rules:
    - trigger: "需求确认"
      nextAgent: "PM"
    - trigger: "PRD 输出完成"
      nextAgent: "PMO"
    - trigger: "任务拆解完成"
      nextAgent: "Architect"
    # ... 其他规则
```

### 2. 信息隔离机制

**一个 Matrix 房间 = 一个项目**：

- 每个项目创建独立的 Matrix 房间
- 相关智能体自动加入房间
- 记忆系统按项目 ID 隔离

### 3. 记忆同步机制

**项目级记忆自动推送**：

- PM 更新 PRD → 自动推送给 PMO、Architect、Frontend、Backend
- Architect 更新接口文档 → 自动推送给 Frontend、Backend
- Security 发现高危漏洞 → 自动推送给 Frontend、Backend 并阻止流程继续

### 4. 消息去重机制

**避免死循环**：

- 每条消息附带唯一的 message_id
- 智能体收到消息后先检查是否已处理
- 避免来回确认导致的死循环

## 五、人机协作边界设计

### 必须人工审核的环节

| 环节 | 审核人 | 理由 |
|------|--------|------|
| PRD 输出 | 产品经理（人） | 需求理解偏差会导致整个项目方向错误 |
| 架构设计 | 技术负责人（人） | 架构错误影响可扩展性和性能 |
| 设计稿 | UI 设计师（人） | 视觉和交互体验需要人工把关 |
| 高危漏洞 | 开发负责人（人） | 安全问题可能导致数据泄露 |
| 测试失败 | 开发团队（人） | 缺陷需要人工分析和修复 |
| 生产部署 | 运维负责人（人） | 部署事故可能影响线上服务 |
| 版本发布 | 项目经理（人） | 发布时机和内容需要人工确认 |

### 可以自动流转的环节

- 任务拆解（PMO）
- 代码生成（Frontend、Backend）
- 代码审查（自动）
- 低危漏洞（标记为"已知风险"）
- 测试通过后的部署（测试环境）
- 文档生成（Delivery）

## 六、踩过的坑

### 坑 1：智能体之间消息"死循环"

**现象**：
- PM 给 PMO 发消息："PRD 已完成"
- PMO 收到后给 PM 发消息："收到，请确认是否需要调整"
- PM 收到后又给 PMO 发消息："已确认，无需调整"
- 无限循环

**解决方案**：
- 每条消息附带唯一的 message_id
- 智能体收到消息后先检查是否已处理
- 设置消息 TTL（Time To Live），超时自动终止

---

### 坑 2：记忆串台

**现象**：
- 项目 A 的需求混到项目 B 里
- Frontend 智能体把项目 A 的组件库用到项目 B

**解决方案**：
- 一个 Matrix 房间 = 一个项目
- 记忆系统按项目 ID 隔离
- 智能体切换房间时自动切换记忆上下文

---

### 坑 3：联调时 Frontend 和 Backend 理解不一致

**现象**：
- Frontend 调用接口时参数格式不对
- Backend 返回的数据结构与接口文档不符

**解决方案**：
- Architect 输出的接口文档作为"契约"
- Frontend 和 Backend 都必须严格遵守接口文档
- 联调时自动验证接口契约一致性
- 不一致时自动报错并阻止流程继续

## 七、当前状态与待验证疑问

### 当前状态

- ✅ 基本流程能跑通
- ✅ 9 个智能体已配置完成
- ✅ 串行协作已实现
- ⚠️ 协作效率还有提升空间
- ⚠️ 并行协作还在研究中（比如 Frontend 和 Backend 同时开发）

### 待验证疑问

1. **不同行业的协作流程差异很大，如何适配？**
   - 当前设计基于产研团队，其他行业可能需要调整

2. **如果某个智能体卡住了，整个流程会不会停摆？**
   - 需要设计"超时机制"和"降级方案"

3. **并行协作如何实现？**
   - Frontend 和 Backend 能不能同时开发？
   - 需要接口契约先行，但联调时可能会有冲突

4. **智能体误操作导致生产事故怎么办？**
   - 需要设计"回滚机制"和"应急响应预案"
